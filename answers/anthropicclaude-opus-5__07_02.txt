## Added tasks

| Task ID | Name | Notes |
|---|---|---|
| `Task_VerifyIdentity` | "Verify Identity" | New task inserted between `StartEvent_1` and `Task_ReviewApplication` (v1 flowed Start → Review Application directly). |
| `Task_RequestDocs` | "Request Additional Documents" | New third outgoing branch from `Gateway_Decision` (flow label "missing docs"), then → `EndEvent_1`. |
| `Task_FinalReject` | "Final Reject" | New task on the "ineligible" branch → `EndEvent_1`. Functionally it occupies the position of the deleted `Task_RejectLoan`, but the element ID is new, so it is counted as an addition rather than a rename. |

## Removed tasks

| Task ID | Name | Notes |
|---|---|---|
| `Task_RejectLoan` | "Reject Loan" | Element (and its flows `Flow_4`/`Flow_6` from v1) no longer exists; its rejection path is now served by the newly added `Task_FinalReject`. |

Retained (not removed): `Task_ReviewApplication` and `Task_ApproveLoan` — the latter is only re-labelled "Approve Loan" → "Auto-Approve Loan" (cosmetic, same ID, ignored per instructions).

## Gateway-type changes

| Gateway ID | v1 type | v2 type | Notes |
|---|---|---|---|
| `Gateway_Decision` | `exclusiveGateway` ("Eligible?") | `inclusiveGateway` ("Eligibility Check") | Exclusive → Inclusive. Semantics change from exactly-one path to one-or-more concurrent paths; branch count grows from 2 (yes/no) to 3 (eligible / missing docs / ineligible). The name change itself is cosmetic and ignored. No merging/join gateway was added in either version (all branches still converge directly on `EndEvent_1`), so the inclusive split remains unbalanced.
